\chapter{Realisierungsphase}

Die Realisierungsphase wurde primär unter dem Aspekt der Kosten- und Zeitoptimierung durchgeführt. Anstatt die Netzwerkinfrastruktur zeitaufwendig über die grafische Benutzeroberfläche (GCP Console) manuell zu konfigurieren, wurde die Bereitstellung in eine Skript-Vorbereitungsphase und eine automatisierte Ausführungsphase unterteilt.

Dieses Vorgehen ermöglichte es, kostenintensive Cloud-Ressourcen (wie HA-VPN-Gateways) erst unmittelbar zum Testzeitpunkt bereitzustellen und nach erfolgreicher Validierung wieder abzubauen. 

\section{Vorbereitungsphase (Skripting und Variablen)}
Das gesamte Deployment wurde in Form von \texttt{gcloud}-Befehlen für das Command Line Interface (CLI) vorbereitet. Um manuelle Eingabefehler zu vermeiden, wurden Netzwerke, Regionen und IP-Adressen als globale Umgebungsvariablen definiert. 

Auszug der initialen Variablen-Deklaration:

\begin{lstlisting}[language=bash, caption={Definition der Umgebungsvariablen}]
# Projekt- und Regionsparameter
export PROJECT_ID="PROJEKT-ID"
export REGION="europe-west3"

# Cloud-Umgebung (Hub)
export CLOUD_VPC="vpc-europe"
export CLOUD_SUBNET="subnet-europe"
export CLOUD_CIDR="10.10.0.0/24"
export CLOUD_ROUTER="cr-europe"
export CLOUD_ASN="65001"
export CLOUD_GW="vpngw-europe"

# Zweigstellen-Umgebung (On-Premises / Spoke)
export BRANCH_VPC="vpc-branch-a"
export BRANCH_SUBNET="subnet-branch-a"
export BRANCH_CIDR="192.168.1.0/24"
export BRANCH_ROUTER="cr-branch-a"
export BRANCH_ASN="65002"
export BRANCH_GW="vpngw-branch-a"

# NCC Hub & HA-VPN Shared Secret
export HUB_NAME="ncc-hub-global"
export SPOKE_NAME="spoke-branch-a"
export SHARED_SECRET="secure-ikev2-key-2026"

# Tunnel- und BGP-Parameter (Auszug Tunnel 0)
export TUN_EU_TO_BR_0="tunnel-eu-to-br-0"
export EU_BGP_0_IP="169.254.1.1"
export BR_BGP_0_IP="169.254.1.2"
\end{lstlisting}

\section{Ausführungsphase (One-Pass Deployment)}
Durch die Vorbereitung der Skripte ließ sich die gesamte Zielarchitektur in der vorkonzipierten Reihenfolge fehlerfrei in der Cloud implementieren.

\subsection{VPCs, Subnetze und Firewall}
Im ersten Schritt wurden die Basisnetzwerke im Custom-Subnet-Modus erstellt. Die Google Cloud Firewall wurde nach dem \textit{Default-Deny}-Prinzip konfiguriert, sodass ausschließlich explizit erlaubter interner Traffic (SSH und ICMP für Ping-Tests) sowie Zugriffe über den Identity-Aware Proxy (IAP) zugelassen werden.

\begin{lstlisting}[language=bash, caption={Erstellung der Basisnetzwerke und Firewall-Regeln}]
# Basis-VPCs anlegen
gcloud compute networks create "${CLOUD_VPC}" --subnet-mode=custom
gcloud compute networks create "${BRANCH_VPC}" --subnet-mode=custom

# Subnetze anlegen
gcloud compute networks subnets create "${CLOUD_SUBNET}" \
  --network="${CLOUD_VPC}" --region="${REGION}" --range="${CLOUD_CIDR}"

gcloud compute networks subnets create "${BRANCH_SUBNET}" \
  --network="${BRANCH_VPC}" --region="${REGION}" --range="${BRANCH_CIDR}"

# Firewall-Regel Cloud (Erlaubt Traffic von Branch & IAP)
gcloud compute firewall-rules create "fw-allow-internal-eu" \
  --network="${CLOUD_VPC}" --allow tcp:22,icmp \
  --source-ranges="${BRANCH_CIDR},35.235.240.0/20"

# Firewall-Regel Branch (Erlaubt Traffic von Cloud & IAP)
gcloud compute firewall-rules create "fw-allow-internal-branch" \
  --network="${BRANCH_VPC}" --allow tcp:22,icmp \
  --source-ranges="${CLOUD_CIDR},35.235.240.0/20"
\end{lstlisting}

\subsection{HA-VPN und BGP-Routing}
Anschließend wurden die Cloud Router und HA-VPN-Gateways bereitgestellt. Da die Konfiguration der lokalen VPN-Endpunkte (z.\,B. Fortinet) produktiv durch den Kunden erfolgt, wurde die Gegenseite hier zu Testzwecken simuliert. Der Code-Auszug zeigt die Gateway-Erstellung und die Cloud-seitigen Tunnel (Branch-Konfiguration siehe Betriebshandbuch).

\begin{lstlisting}[language=bash, caption={Konfiguration der Gateways und BGP-Sitzungen (Auszug)}]
# 1. Cloud- und Branch-Router sowie HA-VPN-Gateways anlegen
gcloud compute routers create "${CLOUD_ROUTER}" \
  --region="${REGION}" --network="${CLOUD_VPC}" --asn="${CLOUD_ASN}"

gcloud compute vpn-gateways create "${CLOUD_GW}" \
  --network="${CLOUD_VPC}" --region="${REGION}"

gcloud compute routers create "${BRANCH_ROUTER}" \
  --region="${REGION}" --network="${BRANCH_VPC}" --asn="${BRANCH_ASN}"

gcloud compute vpn-gateways create "${BRANCH_GW}" \
  --network="${BRANCH_VPC}" --region="${REGION}"

# 2. IPsec Tunnel 0 (Interface 0) erstellen und BGP konfigurieren
gcloud compute vpn-tunnels create "${TUN_EU_TO_BR_0}" \
  --peer-gcp-gateway="${BRANCH_GW}" --region="${REGION}" \
  --ike-version=2 --shared-secret="${SHARED_SECRET}" \
  --router="${CLOUD_ROUTER}" --vpn-gateway="${CLOUD_GW}" --interface=0

gcloud compute routers add-interface "${CLOUD_ROUTER}" \
  --interface-name="if-eu-to-br-0" --ip-address="${EU_BGP_0_IP}" \
  --mask-length=30 --vpn-tunnel="${TUN_EU_TO_BR_0}" --region="${REGION}"

gcloud compute routers add-bgp-peer "${CLOUD_ROUTER}" \
  --peer-name="bgp-branch-a-0" --interface="if-eu-to-br-0" \
  --peer-ip-address="${BR_BGP_0_IP}" --peer-asn="${BRANCH_ASN}" --region="${REGION}"

# 3. IPsec Tunnel 1 (Interface 1 für Hochverfügbarkeit) erstellen
gcloud compute vpn-tunnels create "${TUN_EU_TO_BR_1}" \
  --peer-gcp-gateway="${BRANCH_GW}" --region="${REGION}" \
  --ike-version=2 --shared-secret="${SHARED_SECRET}" \
  --router="${CLOUD_ROUTER}" --vpn-gateway="${CLOUD_GW}" --interface=1

gcloud compute routers add-interface "${CLOUD_ROUTER}" \
  --interface-name="if-eu-to-br-1" --ip-address="${EU_BGP_1_IP}" \
  --mask-length=30 --vpn-tunnel="${TUN_EU_TO_BR_1}" --region="${REGION}"

gcloud compute routers add-bgp-peer "${CLOUD_ROUTER}" \
  --peer-name="bgp-branch-a-1" --interface="if-eu-to-br-1" \
  --peer-ip-address="${BR_BGP_1_IP}" --peer-asn="${BRANCH_ASN}" --region="${REGION}"
\end{lstlisting}

\subsection{Network Connectivity Center (NCC)}
Den Kern der Hub-and-Spoke-Architektur bildet das Network Connectivity Center. Der Hub wurde als zentrale Verwaltungsinstanz angelegt. Im Anschluss wurden die zuvor erstellten VPN-Tunnel mittels der \texttt{linked-vpn-tunnels}-Syntax als \textit{Spoke} an den Hub angebunden, um den nahtlosen Datentransfer zwischen den Netzwerken zu aktivieren.

\begin{lstlisting}[language=bash, caption={Bereitstellung des NCC Hubs und der Spokes}]
# Zentralen Hub erstellen
gcloud network-connectivity hubs create "${HUB_NAME}" \
  --description="Zentraler Hybrid Cloud Hub"

# API-Basis-Pfad zur besseren Lesbarkeit auslagern
export API_BASE="https://www.googleapis.com/compute/v1/projects"
export REGION_PATH="${API_BASE}/${PROJECT_ID}/regions/${REGION}"

# Tunnel-URIs zusammensetzen
export HUB_URI="https://networkconnectivity.googleapis.com/v1/projects/${PROJECT_ID}/locations/global/hubs/${HUB_NAME}"
export TUN0_URI="${REGION_PATH}/vpnTunnels/${TUN_EU_TO_BR_0}"
export TUN1_URI="${REGION_PATH}/vpnTunnels/${TUN_EU_TO_BR_1}"

# VPN-Tunnel als Spoke an den Hub anbinden
gcloud network-connectivity spokes linked-vpn-tunnels create "${SPOKE_NAME}" \
  --region="${REGION}" \
  --hub="${HUB_URI}" \
  --vpn-tunnels="${TUN0_URI},${TUN1_URI}" \
  --site-to-site-data-transfer
\end{lstlisting}

Durch dieses vorstrukturierte, \textit{Infrastructure-as-Code}-ähnliche Vorgehen dauert die aktive Bereitstellung der gesamten komplexen Netzwerkarchitektur weniger als 30 Minuten, was zu einer signifikanten Einsparung im Projektbudget führt.
